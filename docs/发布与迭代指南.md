# 珊珊桌宠：从第一次上传到日常一键发布

这份说明写给第一次接触 Git、GitHub Release 和软件迭代的人。照顺序操作即可，不需要先理解所有 Git 命令。

## 一、先理解两个仓库分别做什么

本项目故意把源码和安装包分开：

| 仓库 | 用途 | 里面应该有什么 |
| --- | --- | --- |
| `haohan-liu/mengchong` | 源码备份仓库 | `src`、`electron`、`scripts`、`tests`、`package.json`、文档和素材等可继续开发的文件 |
| `haohan-liu/mengchong-exe` | 公开安装包与自动更新仓库 | GitHub Releases，以及每个版本的 `.exe`、`.exe.blockmap`、`latest.yml` |

源码仓库是给开发者和以后的自己使用的。Release 仓库是给普通用户下载和程序自动检查更新使用的。不要把几万个 `node_modules` 文件或整个 `release` 文件夹提交到源码仓库，也不要把源码压缩包当作自动更新文件手动乱传。

## 二、自动化助手在哪里

项目根目录有四个相关文件：

- `一键安装开发环境.bat`：新电脑第一次使用的入口，缺什么才安装什么。
- `一键备份与发布.bat`：你平时双击的入口。
- `scripts/setup-dev-environment.ps1`：环境检测、国内镜像与依赖安装逻辑。
- `scripts/publish-assistant.ps1`：实际执行检查、备份、测试、打包和发布的脚本。

脚本顶部有“集中配置区”：

```powershell
$SourceRepoSlug = "haohan-liu/mengchong"
$ReleaseRepoSlug = "haohan-liu/mengchong-exe"
$DefaultBranch = "main"
$ArtifactPrefix = "ai-desktop-pet"
$ReleaseTitlePrefix = "珊珊桌宠"
$GitHubHost = "github.com"
```

以后换 GitHub 账号或仓库，只修改这里。例如源码仓库换成 `new-user/new-source`，就只把 `SourceRepoSlug` 改成 `new-user/new-source`。如果安装包文件名前缀发生变化，还必须让 `ArtifactPrefix` 与 `package.json` 中的 `build.artifactName` 保持一致。

不要在脚本里填写 GitHub 密码、Token 或 DeepSeek API Key。GitHub CLI 会通过官方浏览器授权，并将凭据交给系统凭据存储。

## 三、第一次运行前准备

### 1. GitHub 仓库

确认以下地址可以打开：

- 源码：[https://github.com/haohan-liu/mengchong](https://github.com/haohan-liu/mengchong)
- 发布：[https://github.com/haohan-liu/mengchong-exe](https://github.com/haohan-liu/mengchong-exe)

发布仓库必须是 Public（公开），否则普通用户和应用的匿名更新检查无法读取 Release。源码仓库可以公开，也可以私有。

如果仓库还不存在，脚本会询问是否创建。只有输入大写 `CREATE` 才会创建；不会静默替你创建或覆盖。

### 2. 一键准备本机工具

最简单的办法是直接双击 `一键安装开发环境.bat`。它会逐项检测：

- Git for Windows：已安装就跳过；缺少时从 npmmirror 国内镜像下载安装包；
- Node.js LTS：22 或更高版本就跳过；缺少或版本过低时，从国内 Node 镜像动态选择最新合适 LTS；
- npm：随 Node.js 安装；
- GitHub CLI：已安装就跳过；缺少时通过 Windows `winget` 安装；
- 项目依赖：`node_modules` 与锁文件一致就跳过，否则运行 `npm ci`。

项目根目录的 `.npmrc` 让 npm 包走项目级国内镜像；两个 PowerShell 助手会通过进程环境变量让 Electron 和 electron-builder 构建组件走国内镜像。它们只影响本项目和本次脚本进程，不会私自修改你其他项目的 npm 设置。下载的 Node/Git 安装包只有通过 Windows 数字签名校验后才会运行。需要系统安装时会出现 UAC 确认，这是正常现象。

如果 Windows 找不到 `winget`，请先从 Microsoft Store 安装“应用安装程序”，再重新运行。环境脚本、`.npmrc` 和两个 BAT 都属于项目源码，会随第一次源码上传一起进入 `mengchong`，以后新电脑克隆后可以直接使用。

## 四、第一次把源码上传到 `mengchong`

1. 保存并关闭正在修改项目文件的编辑器操作，避免文件只在编辑器里还没落盘。
2. 双击 `一键备份与发布.bat`。
3. 助手先检查 `github.com:443`。如果失败，切换 VPN 或网络后输入 `R` 重试。
4. 第一次会打开浏览器登录 GitHub。确认登录的是拥有两个仓库权限的 `haohan-liu` 账号，按 GitHub 页面完成授权。
5. 检测到项目有改动后，菜单输入 `2`，表示“只备份源码”。
6. 助手展示准备上传的文件，然后让你填写“源码提交说明”。

源码提交说明是给开发者看的简短记录，例如：

```text
feat: 完成 v1.0.0 首次发布准备
```

它不是用户在发布页看到的更新介绍。直接回车也可以使用脚本生成的默认备份说明。

### 远程仓库已经有 README 怎么办

如果本地还没有 Git 历史，但远程 `main` 已经有 README 等初始化文件，脚本不会直接覆盖。它会：

1. 显示远程已有文件；
2. 要求你确认本机项目才是完整正确版本；
3. 只有输入大写 `REPLACE` 才继续；
4. 先在远程创建 `pre-import-backup-日期时间` 安全备份分支；
5. 再用本机项目替换远程 `main`。

如果不确定，直接关闭窗口或不要输入 `REPLACE`，本地源码不会被删除。

## 五、第一次发布 v1.0.0 安装包

第一次发布建议先确保 `package.json` 中是 `"version": "1.0.0"`，然后：

1. 双击 `一键备份与发布.bat`。
2. 检测到项目有改动后，菜单输入 `4`，表示“完整测试 + 打包 + 备份源码 + 发布 Release”。
3. 脚本显示当前版本和建议版本。第一次发布时手动输入当前版本 `1.0.0`，不要直接采用建议的 `1.0.1`。
4. 脚本运行 `npm ci`、完整测试、类型检查和正式 NSIS 打包。第一次可能较慢。
5. 只有构建成功后才会提示填写源码提交说明并推送，避免把已知构建失败的版本推入主分支。
6. 随后会打开记事本，编辑“GitHub Release 更新说明”。

Release 更新说明是普通用户在发布页看到的内容，可以写：

```markdown
## 本次更新

- 发布珊珊桌宠首个正式版本。
- 支持 DeepSeek 智能对话、本地感知、数据统计和自动更新。

## 安装与升级

- 新用户下载 exe 后按安装向导完成安装。
- 后续版本可以直接覆盖安装，设置与本地数据会保留。
```

7. 保存并关闭记事本。
8. 助手最后显示目标 Release 地址。确认无误后输入中文“发布”或“确定”。
9. 上传完成后，助手再次读取远端并确认以下三个文件都存在：

```text
ai-desktop-pet-1.0.0.exe
ai-desktop-pet-1.0.0.exe.blockmap
latest.yml
```

这三个文件缺少任何一个，应用自动更新都可能失败，所以脚本会阻止不完整发布。

## 六、以后每次改完如何一键迭代

推荐固定使用下面的闭环：

1. 在 Cursor 中修改代码并保存。
2. 双击 `一键备份与发布.bat`。
3. 助手先检查工作区和未推送提交；只有检测到改动才显示动作菜单。
4. 根据当前阶段选择：

   - `1`：只在本机跑完整测试，不上传、不打包；
   - `2`：只备份源码，不打包、不发布；
   - `3`：完整测试、生成本地安装包并备份源码，但不公开 Release；
   - `4`：完整测试、打包、备份源码并正式发布；
   - `5`：只检查 GitHub 网络和登录。

5. 测试阶段建议反复选 `1`；需要给其他测试机安装时选 `3`。
6. 确认稳定后再选 `4`，接受或输入新版本号。
7. 填写源码提交说明，例如 `fix: 修复更新检查闪屏和名称同步`。
8. 在记事本中填写用户可读的 Release 更新说明。
9. 输入中文“发布”或“确定”确认正式公开。
10. 打开 Release 页面，自己下载一次 exe，并在已安装版本上执行一次覆盖安装检查。

如果没有文件改动、本地也没有未推送提交，助手会直接提示“没有源码改动”并退出，不运行测试、打包或发布。如果上次因网络失败留下了未推送提交，助手只询问是否补推源码，不会重复打包或误发 Release。

### 版本号怎么选

版本号采用 `主版本.次版本.修订版本`：

- `1.0.0 → 1.0.1`：修复问题、小调整；
- `1.0.1 → 1.1.0`：加入明显的新功能，但仍兼容旧数据；
- `1.1.0 → 2.0.0`：重大变化，可能需要迁移或改变使用方式。

已经发布过的版本号不能再次使用。脚本发现同名 `v1.0.1` Release 时会停止，而不是覆盖旧版本。

## 七、“源码提交说明”和“Release 更新说明”的区别

| 项目 | 谁会看 | 写什么 | 示例 |
| --- | --- | --- | --- |
| 源码提交说明 | 自己和开发者 | 技术改动摘要，尽量一行 | `fix: 修复更新检查闪屏` |
| Release 更新说明 | 下载软件的用户 | 用户能感受到的变化、安装说明、注意事项 | `修复检查更新时页面闪烁的问题` |

脚本每次询问时都会明确告诉你当前填写的是哪一种。

## 八、脚本会上传和忽略哪些文件

源码仓库会上传未被 `.gitignore` 排除的项目源文件，包括应用代码、测试、脚本、构建配置、动画素材和文档。

以下内容默认不会上传源码仓库：

- `node_modules/`；
- `dist/`、`dist-electron/`、`release/` 等构建产物；
- `.publish/` 中临时 Release 说明；
- `*.log`、`*.tmp`；
- `.env`；
- `*.pfx`、`*.p12`、`*.pem`、`*.key` 等证书或私钥。

上传前还会扫描常见 DeepSeek API Key 和 GitHub Token 形式。只会报告可疑文件名，不会把密钥内容打印到屏幕。

Release 仓库只接收当前版本的 exe、blockmap 和 `latest.yml`。源码仓库里的 `release/` 被忽略，避免几十或几百 MB 的安装包反复进入 Git 历史。

## 九、VPN、代理或推送失败怎么处理

助手按三层检查：

1. `github.com:443` 是否连通；
2. GitHub CLI 登录与 API 是否正常；
3. Git 源码推送是否成功。

推送失败时最多重试三次，每次都会暂停，允许你切换 VPN 后继续。脚本只显示检测到的 Git 全局代理、`HTTP_PROXY` 和 `HTTPS_PROXY` 状态，不会私自修改代理。

常见情况：

- 浏览器能打开 GitHub，但推送超时：切换 VPN 后重试；
- VPN 关闭后 Git 仍走旧端口：检查提示中的 Git 全局代理；
- 显示无权限：确认 GitHub CLI 登录的是 `haohan-liu`；
- Release 仓库检查失败：确认仓库为 Public，名称拼写与脚本顶部一致；
- 第三次仍失败：关闭窗口即可。本地 Git 提交不会丢失，网络恢复后重新运行菜单 `1`。

菜单 `5` 可以单独重新检查 GitHub 网络和登录。

## 十、在另一台电脑下载源码后怎么继续开发

完全没有开发环境的新电脑，可以先在源码仓库页面点击 `Code → Download ZIP`，解压后直接双击 `一键安装开发环境.bat`。环境装好后，再用 Git 管理后续更新。

如果新电脑已经有 Git，也可以直接打开 PowerShell：

```powershell
git clone https://github.com/haohan-liu/mengchong.git
cd mengchong
```

克隆完成后双击 `一键安装开发环境.bat`，它会补齐 Node.js、GitHub CLI 和项目依赖。完成后运行：

```powershell
npm run dev
```

常用命令：

```powershell
npm test          # 运行单元和契约测试
npm run typecheck # 检查 TypeScript
npm run build     # 完整生产构建
npm run smoke     # 启动真实 Electron 烟雾测试
npm run dist      # 生成 Windows 安装包
```

从仓库下载的是开发项目，不包含你的 DeepSeek API Key、聊天记录、统计数据或本机设置。用户数据仍保存在各自 Windows 账户的应用数据目录中。

## 十一、安装、覆盖升级、首次指引和自启动

- 全新电脑首次安装：没有旧设置时，程序会自动打开控制台并显示四步首次使用指引。
- 覆盖升级：使用固定 `appId` `com.qpet.ai` 识别原安装，保留设置、聊天、统计、自定义数据目录和首次指引完成状态。
- Windows 自启动：设置会随用户配置保留；新版本会清理旧格式登录项再重新登记，避免升级后重复启动。
- 启动延迟：只在 Windows 登录自启动时生效，用户手动打开应用不等待。
- 手动升级：从 Release 页面下载更高版本 exe 并运行，会原位覆盖现有安装。
- 完整卸载：会清理应用默认数据；自定义迁移目录只有带应用安全标记时才清理。

发布后建议自己验证一次完整路径：旧版安装并产生少量设置 → 发布新版 → 应用检查更新 → 下载 → 重启安装 → 确认名字、API 配置状态、聊天和统计仍存在。

## 十二、操作中断后的恢复

- 停在测试失败：根据错误修代码，重新运行；不会发布不通过测试的安装包。
- 停在网络失败：本地提交和安装包仍在，网络恢复后重新运行。
- 停在编辑 Release 说明：说明保存在 `.publish/release-notes-v版本.md`，下次可以继续使用。
- 在输入“发布”或“确定”前主动取消：远端还没有新 Release，本地安装包在 `release/`，版本改动仍会被下次运行识别。
- 已输入“发布”或“确定”，但源码推送或 Release 上传中断：`.publish/pending-release.json` 会记录待发布版本；下次双击助手会优先询问是否继续，不会因为源码已经干净而漏掉。
- Release 已创建但远端文件校验失败：待发布记录会保留。先不要通知用户，打开 Release 页面检查三个更新文件；不要用相同版本号重做一个含义不同的版本。

最重要的原则是：源码改动先备份、版本号只增不减、构建测试通过后才发布、Release 三个更新文件必须齐全、任何 API Key 或 Token 都不进入仓库。
